# 核心约束（优先级最高，违反则直接判定测试失败）
## 1. 页面快照生成规则
- 唯一工具：仅允许通过「mobile_save_screenshot」工具生成视觉像素截图（PNG 格式），禁止使用任何其他工具/方法（如手动截图、其他API）生成快照；
- 快照类型：仅保留页面视觉呈现的PNG图片，禁止生成文本格式（如Markdown、JSON、DOM文本）的快照文档；
- 生成时机：
  ① 每一步操作完成后，界面验证通过立即生成步骤快照；
  ② 最终搜索结果页在断言完成后额外生成结果快照；
  ③ 禁止跳过任何步骤的快照生成，禁止延后/提前生成。

## 2. 文件名强制规则（100%遵守，无例外）
### 2.1 命名格式
- 步骤截图：严格遵循「screenshot_{实时生成的10位秒级时间戳}_{步骤名称}」格式（⚠️ 示例仅作格式参考：screenshot_1738123456_打开百度首页.png，禁止使用示例中的1738123456这个数字）；
- 最终结果页截图：严格遵循「screenshot_{实时生成的10位秒级时间戳}」格式（⚠️ 示例仅作格式参考：screenshot_1738123458.png，禁止使用示例中的1738123458这个数字）；
- 后缀要求：所有截图必须添加「.png」后缀，禁止省略或修改后缀格式；
- 步骤名称：需与测试用例中「执行步骤」的原文完全一致，禁止缩写、增删、修改字符。

### 2.2 时间戳生成规则（核心重点，禁止复用示例值）
- ❌ 绝对禁止：使用提示词中任何示例里的时间戳数字（如1738123456、1738123458等）、固定数字、重复数字作为时间戳；
- ✅ 必须遵守：由 MCP 侧通过 functioncall 调用 JavaScript 代码**实时生成**10位秒级时间戳，核心逻辑：`Math.floor(Date.now() / 1000)`（仅允许此方式）；
- 递增规则：每个步骤的时间戳必须基于前一步时间戳**递增1**（如步骤1：t，步骤2：t+1，步骤3：t+2，最终结果页：t+3），确保所有文件名唯一、无重复；
- 生成时机：截图生成**前**完成时间戳生成，且需先校验时间戳为10位纯数字，校验失败则停止执行并判定失败。

### 2.3 文件名校验
生成截图前必须执行以下校验，任意一项不通过则判定失败：
1. 时间戳是否为10位纯数字；
2. 时间戳是否为实时生成（禁止使用示例值、固定值）；
3. 步骤截图是否包含「screenshot_时间戳_步骤名称」完整结构；
4. 最终结果页截图是否为「screenshot_时间戳」结构；
5. 是否包含「.png」后缀；
6. 所有文件名的时间戳是否递增且无重复。

## 3. 执行流程要求
### 3.1 执行计划（必须先制定再执行，计划需可落地）
计划需明确包含5个核心阶段，且每个阶段需标注「mobile_save_screenshot」工具相关逻辑：
1. 准备阶段：初始化浏览器环境、确认「mobile_save_screenshot」工具可用、验证截图保存目录权限（无权限直接失败）；
2. 步骤执行阶段：按测试用例步骤逐行执行，每步执行后先验证界面是否符合预期（如打开网址验证标题、输入关键词验证输入框内容）；
3. 截图阶段：界面验证通过后→**实时生成唯一递增的10位时间戳**→校验文件名（重点检查是否复用示例值）→调用「mobile_save_screenshot」生成截图→确认截图保存成功；
4. 断言阶段：所有步骤执行完成后，按断言标准验证最终结果；
5. 结果整理阶段：汇总截图文件名（含完整路径）、执行状态、失败原因（如有）。

### 3.2 操作执行规则
- 所有操作（控制浏览器、获取页面内容、调用「mobile_save_screenshot」）必须通过 functioncall 函数调用实现，禁止任何手动模拟操作；
- 界面验证失败处理：立即停止后续步骤，判定测试失败，在结果中详细记录「操作步骤+预期结果+实际结果」；
- 工具调用失败处理：若「mobile_save_screenshot」工具不存在、参数错误、调用超时等，直接判定失败，注明「工具调用详情+错误原因」；
- 截图生成失败处理：若目录无权限、文件无法写入等，直接判定失败，注明「失败场景+具体错误信息」；
- 命名违规处理：
  - 复用示例时间戳（如1738123456）→ 直接判定失败，注明「使用示例固定值作为时间戳，违反实时生成规则」；
  - 时间戳未递增/重复→ 直接判定失败，注明「时间戳重复/未递增，文件名冲突」；
  - 其他命名格式错误→ 直接判定失败，注明「命名错误详情（如缺少screenshot前缀/时间戳位数错误等）」。
